# Starlight 项目级代码审查与清理计划

## 目标

对 `songloft-plugin-starlight` 当前代码做项目级 review，删除已经确认无用的冗余代码，修复有证据支撑的缺陷，并保持插件可构建、可校验、测试通过。

## 当前阶段

阶段 5：验证与交付，状态：进行中。代码清理、缺陷修复、发布 workflow 与 manifest hash 闭环已通过本地验证；提交/推送前需确认最终暂存范围。

## 阶段

| 阶段 | 状态 | 交付内容 |
| --- | --- | --- |
| 1. 审查基线 | 完成 | 当前 git 状态、测试状态、模块清单、已知历史变更确认 |
| 2. 自动化扫描 | 完成 | 残留引用、未使用文件/导出、重复逻辑、TODO、类型/构建/测试问题 |
| 3. 人工代码审查 | 完成 | UI、Songloft 曲库、音乐桥接、下载、MIoT 播放、语音口令、发布元数据重点审查 |
| 4. 清理与缺陷修复 | 完成 | 删除确认冗余代码，修复可复现或可证明缺陷，补充/更新测试 |
| 5. 验证与交付 | 进行中 | `npm test`、`npm run typecheck`、`npm run build`、`npm run validate`、版本戳、提交和推送 |

## 验收标准

- 当前插件目录工作树仅包含本任务相关变更；父目录既有未跟踪文件不纳入提交。
- 没有本地播放器残留可见控件、导入、事件绑定或 CSS 选择器。
- 删除的代码必须有引用扫描证据支持，不能删除仍被运行时或测试使用的功能。
- 每个修复的缺陷都有根因记录和验证命令。
- 保留用户明确要求的核心功能：音箱控制、Songloft 曲库、LX 音源搜索/歌单/排行、下载源独立、语音口令、插件源版本规则。
- 全量测试、类型检查、构建、插件校验通过。

## 风险

- `static/js/music.js` 仍是最大前端文件，清理时要避免顺手大重构导致回归。
- 宿主 Songloft API 形态依赖运行环境，涉及接口行为的修复需要优先依赖现有测试和封装边界。
- JS 静态文件不受 TypeScript 完整未使用检查保护，需要结合 import 扫描、测试和构建结果判断。
- 发布链路已用 `release:manifest` 和 workflow 测试闭环；推送后 workflow 会再次按实际时间戳更新版本并提交 release metadata。
