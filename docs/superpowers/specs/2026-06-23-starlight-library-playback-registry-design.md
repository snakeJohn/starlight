# Starlight 曲库、播放、列表与发布设计

## 背景

Starlight 现有能力已经覆盖 MIoT 音箱账号、设备、播放控制、语音口令、LX 音源搜索、歌单、排行、自建歌单、下载音源和下载任务。本次需求在现有功能上继续增强列表操作、Songloft 曲库接入、失败换源、本插件全局播放控件、移动端兼容和插件源发布流程。

本设计只描述要实现的行为和边界。后续实施必须遵守 TDD：先写失败测试，再改实现。

## 目标

1. 搜索、歌单、排行列表统一每页 20 条，并在长列表中使用滚动容器。
2. 搜索结果增加清空、复选框和批量导入能力。
3. 歌单详情、我的歌单展开详情、排行歌曲详情都支持移动端友好的分页与滚动。
4. 新增 Songloft 歌曲库、歌单、本地歌曲读取能力，并能通过语音口令推送到智能音箱。
5. 播放和下载失败时自动切换音源，播放源与下载源严格隔离。
6. 增加本插件自己的全局播放控件，用于本地插件播放和音箱播放状态管理，不修改 `songloft-player`。
7. 按 Songloft 插件源方式一，在 GitHub 仓库内提供 Raw URL 插件源。
8. 版本统一为 `V-yyyy.mm.dd.hh.mm`，每次提交 GitHub 前按实际构建时间自动更新。

## 非目标

1. 不修改 `songloft-player` 源码。
2. 不预置任何 LX 音源或下载音源。
3. 不把播放音源用于下载，也不把下载音源用于播放。
4. 不做大规模 UI 视觉重设计。本次只做布局、滚动、移动端兼容和控件补齐。

## 推荐方案

采用分阶段增量实现。

第一阶段做 UI 列表机制：分页、滚动、清空、复选框、批量操作。

第二阶段做 Songloft 数据接入：歌曲库、歌单、本地歌曲、自身后端封装接口，以及语音口令匹配。

第三阶段做失败换源：播放链路和下载链路分别增加候选音源重试。

第四阶段做本插件全局播放控件：保留单条歌曲 `播放` 按钮，按钮行为改为加入或推送到本插件本地全局播放队列；音箱播放仍使用 `音箱` 按钮。

第五阶段做插件源和版本自动化：`registry.json`、`plugin.json` 发布字段、版本脚本、测试校验。

这种方案每一步都能独立验证，降低跨 UI、后端、语音和发布流程同时变更带来的风险。

## 搜索菜单设计

搜索结果固定每页 20 条。

结果区新增 `清空` 按钮。点击后清空关键词、搜索结果、分页、搜索总数和已选择歌曲。

搜索结果列表使用内部滚动容器，避免大量结果撑高页面。

每条搜索结果增加复选框，支持批量选择。

单条歌曲继续保留 `播放` 按钮。全局播放控件完成后，`播放` 按钮用于推送到本插件本地播放队列，由本插件全局播放控件管理播放。它不再只是临时导入后触发 Songloft 原生底部播放器。

单条歌曲继续保留：

- `播放`
- `导入 Songloft 歌曲库`
- `下载`
- `音箱`
- `加入歌单`

搜索结果新增批量工具条：

- `全选当前页`
- `取消选择`
- `批量导入 Songloft 歌曲库`
- `批量加入目标歌单`
- `批量下载`
- `批量音箱播放`

批量加入歌单使用“我的歌单”中的 `加入目标歌单` 下拉框作为目标。

## 歌单菜单设计

歌单搜索结果固定每页 20 条，并放入滚动容器。

搜索出来的歌单继续支持收藏，收藏后加入“我的歌单”。

搜索歌单的详情列表固定每页 20 条，并放入滚动容器。分页保留上一页、当前页、下一页、指定页。

“我的歌单”展开详情固定每页 20 条，并放入滚动容器。点击 `查看歌曲` 展开，再点击 `正在查看` 收起。

“我的歌单”展开后的歌曲支持复选框和批量操作：

- 批量加入目标自建歌单
- 批量下载
- 批量音箱播放

导入歌单仍位于“我的歌单”和“搜索歌单”之间。

导入歌单保存歌曲时以“歌曲名 + 歌手”为核心数据，不依赖平台歌曲 ID。播放导入歌单时，再根据“歌曲名 + 歌手”跨 5 平台匹配可播放 URL，减少平台歌曲失效造成的影响。

已导入歌单继续支持：

- 查看歌曲
- 同步 Songloft 歌单
- 播放歌单
- 音箱播放
- 选中歌曲加入自建歌单

## 排行菜单设计

榜单列表宽度保持当前优化方向，避免横向撑破布局。

点击榜单后的歌曲列表固定每页 20 条，并放入滚动容器。原有分页能力不变。

榜单歌曲行继续保留单曲操作：

- 播放
- 导入 Songloft 歌曲库
- 下载
- 音箱
- 加入歌单

## Songloft 数据接入设计

插件新增自身封装接口，前端只调用插件接口，避免前端直接处理 Songloft token：

- `GET /api/songloft/songs`
- `GET /api/songloft/playlists`
- `GET /api/songloft/playlists/:id/songs`
- `GET /api/songloft/local-songs`

后端优先使用 Songloft 插件 SDK：

- `songloft.songs.list()`
- `songloft.playlists.list()`
- `songloft.playlists.getSongs()`

必要时通过宿主 API 使用 `/songs`、`/playlists`、`/playlists/{id}/songs`。Songloft 实际宿主会在外层带 `/api/v1` 前缀，插件内部统一处理。

UI 上新增 Songloft 数据查看能力，可查看：

- Songloft 歌曲库
- Songloft 歌单
- Songloft 歌单歌曲
- 本地歌曲

这些歌曲都支持推送到智能音箱。

## 语音口令设计

语音口令的匹配优先级：

1. 自建歌单
2. Songloft 歌单
3. Songloft 曲库和本地歌曲
4. 在线 5 平台搜索兜底

`播放歌单 xxx` 优先匹配自建歌单，再匹配 Songloft 歌单。

`播放歌曲 xxx` 优先匹配 Songloft 曲库和本地歌曲，再按“歌曲名 + 歌手”走在线搜索。

本地歌曲推送音箱时使用 Songloft 可播放 URL。优先通过歌曲 ID 构造 `/songs/{id}/play` 类型 URL，确保音箱能访问宿主提供的播放地址。

## 播放失败换源设计

播放链路只使用播放音源。

播放失败时：

1. 先尝试歌曲自带 `source_data`。
2. 如果解析 URL 失败或音箱推送失败，按启用播放音源顺序重新搜索“歌曲名 + 歌手”。
3. 找到候选歌曲后尝试解析 URL。
4. 成功后继续播放。
5. 全部候选失败后返回明确错误，包含已尝试音源数量。

自建歌单和导入歌单中的动态歌曲也使用同一套换源逻辑。

## 下载失败换源设计

下载链路只使用下载音源。

下载失败时：

1. 先尝试当前歌曲携带的下载音源数据。
2. 失败后按启用下载音源顺序重新搜索“歌曲名 + 歌手”。
3. 找到候选歌曲后尝试解析下载 URL。
4. 成功后导入为当前插件来源的远程歌曲，再调用 Songloft 下载接口。
5. 全部失败后记录失败原因，批量下载继续处理下一首。

下载失败绝不回退到播放音源。

## 本插件全局播放控件设计

不修改 `songloft-player`。

插件新增自己的全局播放控件。控件用于本插件本地播放队列和音箱播放状态管理，视觉上保持 Songloft 风格。

控件包含：

- 歌曲封面
- 歌曲名称
- 歌手与播放状态
- 上一首
- 暂停播放 / 继续播放
- 下一首
- 停止播放
- 播放模式
- 当前队列或歌单列表

搜索、歌单详情、排行歌曲中的单条 `播放` 按钮保留。全局播放控件完成后，`播放` 按钮把歌曲推送到本插件本地播放队列，并由该控件管理播放。

`音箱` 按钮仍负责推送到 MIoT 智能音箱。这样本地播放和音箱播放职责分离。

## 移动端 UI 设计

移动端只做兼容性优化，不做大幅视觉改版。

适配规则：

- 列表行在窄屏下改为封面、信息、操作三段纵向布局。
- 操作按钮允许换行。
- 批量工具条在窄屏下折行或横向滚动。
- 分页栏在窄屏下分两行展示，指定页输入不能溢出。
- 全局播放控件在窄屏下使用紧凑布局。
- 所有滚动列表设置稳定 `max-height`，避免整页过长。

## 插件源设计

按 Songloft 插件源文档“方式一 仓库内 Raw URL”实现。

仓库根目录新增 `registry.json`：

```json
{
  "name": "Starlight Plugin Registry",
  "plugins": [
    "https://raw.githubusercontent.com/snakeJohn/starlight/main/plugin.json"
  ]
}
```

`plugin.json` 更新发布字段：

- `homepage`: `https://github.com/snakeJohn/starlight`
- `updateUrl`: `https://raw.githubusercontent.com/snakeJohn/starlight/main/plugin.json`
- `download_url`: 指向 GitHub Release zip 包

发布时确保 Raw `plugin.json` 中的 `download_url` 可被 Songloft 读取。

## 版本规则

统一版本格式：

```text
V-yyyy.mm.dd.hh.mm
```

示例：

```text
V-2026.06.23.11.27
```

以下文件必须保持一致：

- `package.json`
- `package-lock.json`
- `plugin.json`
- 发布产物元数据

新增版本脚本，在提交 GitHub 前按实际构建时间更新版本。

版本测试必须校验：

- 版本带 `V-` 前缀。
- 年、月、日、小时、分钟均为定长数字。
- 三份本地元数据版本一致。

## 测试策略

每个阶段都先写失败测试，再实现。

UI 测试覆盖：

- 搜索清空按钮。
- 搜索结果 20 条分页。
- 搜索结果复选框和批量工具条。
- 单条 `播放` 按钮保留，并绑定本插件播放队列行为。
- 歌单、我的歌单、排行详情的 20 条分页与滚动容器。
- 移动端关键布局类存在且不溢出。

后端测试覆盖：

- Songloft 曲库接口。
- Songloft 歌单接口。
- 本地歌曲接口。
- 语音口令匹配 Songloft 歌单和本地歌曲。
- 播放失败后切换播放音源。
- 下载失败后切换下载音源，且不使用播放音源。

发布测试覆盖：

- `registry.json` 格式。
- `plugin.json` 的 Raw URL 字段。
- `V-yyyy.mm.dd.hh.mm` 版本格式。
- `package.json`、`package-lock.json`、`plugin.json` 版本一致。

最终验证命令：

```powershell
npm test
npm run typecheck
npm run build
npm run validate
```

如涉及页面布局，还需要用浏览器检查桌面和移动端视图。
